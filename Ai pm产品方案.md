# Grove — 你的 AI 产品经理

> **Grove** /ɡroʊv/ — 果林。产品从一颗种子成长为一棵大树，Grove 是照料这片林子的人。

---

## 一、产品定位

**一句话描述：** Grove 是一个以独立团队成员身份存在的 AI 产品经理，拥有自己的 GitHub 账号（`@grove-pm`）和飞书身份，主动参与项目管理、需求追踪和进度把控。

**核心理念：** 不是"每个人手里的工具"，而是"团队里的第六个人"。Grove 有自己的名字、自己的 GitHub 身份和飞书账号、自己的工作节奏，团队成员和它协作就像和一个真人 PM 协作一样。它同时活跃在 GitHub（代码与任务）和飞书（日常沟通）两个平台上。

**命名含义：** 产品就像一片果林（Grove）——每个功能模块是一棵树，从种子（idea）生根、抽枝、结果。Grove 是照料这片林子的园丁，它不替你种树，但它确保每棵树得到阳光和水分，整片林子朝着正确的方向生长。

**目标用户：** 5 人左右的小型开发团队，使用 GitHub Issues/Projects 做项目管理，飞书做日常沟通协作。

---

## 二、产品形态推荐

### 推荐方案：GitHub App + 飞书 Bot + 定时任务引擎（双平台并行）

经过对比分析，推荐 **GitHub App + 飞书 Bot** 双平台并行方案：

| 方案 | 优点 | 缺点 | 适合度 |
|------|------|------|--------|
| 仅 GitHub App | 天然集成代码流程 | 团队日常沟通不在 GitHub | ⭐⭐⭐ |
| 仅飞书 Bot | 沟通方便，覆盖全员 | 需要额外集成 GitHub | ⭐⭐⭐ |
| **GitHub App + 飞书 Bot** | **代码管理 + 日常沟通全覆盖，双平台独立身份** | 需要同时维护两个通道 | **⭐⭐⭐⭐⭐** |

**双平台架构的核心思路：**

- **GitHub** 是 AI PM 的"办公桌" — 在这里管理代码、Issue、PR、产品文档
- **飞书** 是 AI PM 的"工位" — 在这里和团队日常沟通、推送报告、接收指令、参与讨论

**为什么需要飞书：**

1. **团队沟通的主阵地** — 开发者不会一直盯着 GitHub，但飞书消息随时能看到
2. **更自然的交互** — 在群里 @Grove 提需求、问进度，比在 GitHub Issue 里写评论更直觉
3. **实时通知** — 风险预警、每日报告推送到飞书群，比 GitHub 通知更及时
4. **非开发成员覆盖** — 设计师、运营等不常用 GitHub 的人也能和 AI PM 协作
5. **飞书生态** — 可以利用飞书文档、飞书多维表格、日历等原生能力

**GitHub 侧职责：**

1. **独立身份** — Bot 账号（`@grove-pm`），能创建 Issue、评论 PR、分配任务
2. **事件驱动** — 监听 PR 提交、Issue 变更等事件，实时响应
3. **定时巡检** — 通过 GitHub Actions 每日自动运行检查
4. **文档管理** — PRD 和需求文档以 Markdown 存储在仓库中

**飞书侧职责：**

1. **独立身份** — 飞书自建应用 Bot，显示为团队中的一个"成员"
2. **群聊交互** — 在项目群中接收指令（@Grove）和推送消息
3. **报告推送** — 每日站会报告、风险预警直接推送到飞书群
4. **文档同步** — 重要产品文档可同步生成飞书文档（利用飞书 OpenAPI）
5. **审批流程** — 需求变更、任务重新分配等可通过飞书审批卡片确认

### 飞书集成技术方案

飞书提供了完善的开放平台能力：

**飞书自建应用（Custom App）：**
- 创建企业自建应用，获得 App ID 和 App Secret
- 配置消息卡片、事件订阅、Bot 能力
- 应用在飞书中显示为一个独立的 Bot 成员

**核心 API 能力：**

| 能力 | 飞书 API | 用途 |
|------|---------|------|
| 发消息 | `POST /im/v1/messages` | 推送报告、通知、预警 |
| 消息卡片 | Interactive Message Card | 可交互的任务卡片（确认/拒绝/评论） |
| 事件订阅 | Event Subscription（WebSocket） | 监听群消息、@提及、反馈 |
| 飞书文档 | Docs API | 创建/更新飞书文档（同步 PRD） |
| 多维表格 | Bitable API | 项目进度看板（可选） |
| 日历 | Calendar API | 里程碑截止日期同步到日历（可选） |

**飞书 WebSocket 长连接：**
飞书采用 WebSocket 长连接模式（非传统 Webhook），服务端维持持久双向连接，无需公网暴露端口，部署更简单。

**消息卡片示例（任务分配确认）：**

```json

  "card": {
    "header": {
      "title": { "tag": "plain_text", "content": "🌳 Grove — 新任务分配" },
      "template": "blue"
    },
    "elements": [
      {
        "tag": "div",
        "text": {
          "tag": "lark_md",
          "content": "**任务：** 实现用户登录页面\n**优先级：** P1\n**关联 Issue：** [#23](github-link)\n**预估工时：** 3 天\n**分配给：** @张三"

          { "tag": "button", "text": { "tag": "plain_text", "content": "🔄 需要调整" }, "value": { "action": "negotiate" } },
          { "tag": "button", "text": { "tag": "plain_text", "content": "❌ 无法承接" }, "type": "danger", "value": { "action": "reject" } }
        ]
      }
    ]
  }
}
```

```

─────────────────────────────────────────────────────

─────────────────────────────────────────────────────

卡片点击 "✅ 接受任务"    →  AI PM  →  GitHub Issue 分配 assignee
```

## 三、核心功能模块

### 模块 1：产品文档生成器（Product Doc Generator）

**核心原则：飞书文档是唯一真实来源（Single Source of Truth）**

产品文档以**飞书文档**为主阵地，所有团队成员和 AI PM 共同维护同一份文档。不再是"AI PM 单方面输出，团队被动查看"，而是一个真正的协作过程。

**为什么选飞书文档而非 GitHub Markdown：**
- 飞书文档支持多人实时协作编辑，所见即所得
- 非技术成员（设计师等）不需要学 Git 就能参与编辑
- 评论、@提及、修改历史等协作功能开箱即用
- AI PM 通过飞书 Docs API 读写文档，和人类编辑体验一致
- 文档变更自动通知所有关注者

**文档架构：**

飞书知识库中创建一个「产品空间」，结构如下：

```
📁 产品空间（飞书知识库）
├── 📄 产品愿景与定位          ← 团队 + AI PM 共同维护
├── 📁 PRD 文档
│   ├── 📄 功能模块 A — PRD    ← 每个功能一份，所有人可编辑
│   ├── 📄 功能模块 B — PRD
│   └── 📄 ...
├── 📄 用户故事总表            ← AI PM 自动生成，团队可调整
├── 📄 里程碑规划              ← AI PM 建议，团队确认
├── 📄 决策记录                ← AI PM 自动记录每次重要变更
└── 📄 变更日志                ← 自动追踪所有文档修改
```

**触发方式：**
- 在飞书群里 @Grove 说 "我有个新想法…"
- 在飞书文档中直接写下初步想法，@Grove 请它补充
- 在 GitHub 创建 Issue + 标签 `product-idea`

**工作流程：**

```
场景 A：从对话到文档（飞书群发起）
─────────────────────────────────────
团队在群里讨论一个想法
       ↓
@Grove "帮我们整理成 PRD"
       ↓
AI PM 在群里提出结构化问题（5-8 个关键问题）
  - 目标用户是谁？
  - 核心解决什么痛点？
  - 与竞品的关键差异？
  - MVP 包含哪些功能？
  - 成功指标是什么？
       ↓
团队在群里逐一回答、讨论
       ↓
AI PM 在飞书知识库中创建 PRD 文档初稿
       ↓
在群里发送文档链接："PRD 初稿已生成，请大家审阅和修改"
       ↓
团队成员直接在文档中编辑、评论
       ↓
AI PM 监听文档变更，响应 @提及，持续完善
       ↓
团队确认 → PRD 定稿

场景 B：从文档到文档（飞书文档内发起）
─────────────────────────────────────
某成员在飞书文档中写下粗略想法
       ↓
在文档中 @Grove "帮我把这个想法展开成完整 PRD"
       ↓
AI PM 直接在同一份文档中补充结构、细节、验收标准
       ↓
其他成员看到通知，进入文档协作编辑
       ↓
AI PM + 团队多轮迭代，直到定稿
```

**AI PM 在文档中的行为规范：**
- AI PM 的编辑带有明确标识（通过飞书 Bot 身份），团队能分辨哪些是 AI 写的
- AI PM 修改已有内容时，会先在评论中说明原因，等待确认后再改
- AI PM 新增内容直接写入，但会在群里通知 "我在 PRD 中补充了 XX 章节，请 review"
- 重大变更（如需求删除、优先级调整）必须通过飞书卡片让相关人确认

**文档 → GitHub 自动同步：**

飞书文档是编辑主阵地，但 GitHub 仓库中保持一份同步副本，用于代码-需求对齐检查：

```
飞书文档变更 → AI PM 检测 → 自动同步到 GitHub /docs/prd/ 目录（Markdown 格式）
                          → 如果涉及已有 Issue，自动更新 Issue 描述
                          → 记录变更到决策日志
```

**输出物：**
- 飞书知识库中的 PRD 文档（主版本，所有人可编辑）
- GitHub `docs/prd/` 中的 Markdown 同步副本（供代码对齐检查使用）
- 飞书文档中的 User Stories 章节（从 PRD 自动拆解）
- 里程碑规划（飞书文档 + GitHub Milestones 双向同步）

### 模块 2：任务拆解与分配（Task Breakdown & Assignment）

**触发方式：** PRD 飞书文档定稿后自动触发，或在飞书群 / GitHub Issue 中 @Grove 请求拆解。

**工作流程：**

```
从飞书知识库读取最新 PRD 文档
       ↓
拆解为 Epics → User Stories → Tasks
       ↓
自动创建 GitHub Issues（带标签、里程碑、优先级）
       ↓
同时更新飞书 PRD 文档中的「用户故事」章节（双向链接到 GitHub Issue）
       ↓
基于团队成员角色 + 历史数据，推荐任务分配
       ↓
飞书群发送任务分配卡片（接受/调整/拒绝）
       ↓
确认后 → GitHub Issue 分配 assignee + 更新 Projects Board
```

**智能分配逻辑：**
- 维护一份 `team.yml` 配置文件，定义每个成员的角色和技能标签
- 分析成员最近的 commit 模式，判断其当前工作负载
- 优先分配给技能匹配且负载较低的成员
- 始终是"建议"而非"强制"，成员可以在 Issue 中回复调整

**team.yml 示例：**

### 模块 3：每日巡检与站会报告（Daily Standup Report）

**触发方式：** 每天早上 9:00（可配置）通过 GitHub Actions 定时触发。

**巡检内容：**

```
1. 活动检查
   - 每个成员昨天的 commit 数量和内容摘要
   - 新提交的 PR 及其状态（open / reviewing / merged）
   - Issue 状态变更（新开 / 关闭 / 进行中）

2. 进度对比
   - 将当前 Issues 完成情况与里程碑时间线对比
   - 计算每个 Epic 的完成百分比
   - 标出"落后于计划"的任务（红色标记）

3. 代码-需求对齐检查
   - 读取最近 24h 的 commit message 和 diff 摘要
   - 与 PRD 中的需求条目匹配
   - 标出"无法关联到任何需求"的代码变更（可能的需求偏离）

4. 风险提醒
   - 超过 3 天无更新的 Issue
   - 超过 48h 无 review 的 PR
   - 里程碑截止日期前 3 天的未完成任务
```

**输出物：** 一个自动创建的 GitHub Issue，标签为 `daily-report`，格式示例：


### 模块 4：PR 审查与需求对齐（PR Review & Alignment Check）

**触发方式：** 每当有新 PR 提交时，自动通过 GitHub Webhook 触发。

**工作流程：**

```
检测到新 PR
       ↓
读取 PR 关联的 Issue（通过 commit message 或 PR 描述中的 #issue_number）
       ↓
从飞书知识库获取关联 Issue 对应的 PRD 章节 / User Story
       ↓
分析 PR 代码变更（diff）
       ↓
对比代码实现与飞书 PRD 中的需求描述
       ↓
在 PR 中发表评论：
  - 需求覆盖度评估（是否完整实现了需求）
  - 遗漏项提醒（需求中要求但代码未实现的部分）
  - 超出范围提醒（代码中有但需求未提及的改动）
```

**注意：** AI PM 只做产品层面的 review（需求对齐），不做代码质量 review（那是 tech lead 的工作）。这保持了角色清晰。

### 模块 5：文档自动反向同步（Code → Doc Auto Sync）

**核心问题：** 现实开发中，代码会不断演进——加了新参数、改了交互流程、调整了数据结构——但产品文档往往跟不上。时间一长，文档和实际产品严重脱节，成为"废纸"。

**AI PM 的解决方式：** 持续监测代码变更，主动发现"代码已经变了但文档没更新"的情况，自动或半自动地更新飞书 PRD 文档。

**触发方式（三种场景）：**

```
场景 A：PR 合并时触发（实时）
──────────────────────────
PR 被合并到主分支
       ↓
AI PM 分析合并的代码变更
       ↓
与飞书 PRD 文档对比，判断是否存在差异
       ↓
┌─ 小幅变更（参数调整、文案修改等）
│    → AI PM 直接更新飞书文档对应章节
│    → 在文档中留下变更备注："基于 PR #45 自动更新"
│    → 飞书群通知："已根据最新代码更新了 PRD 中的「登录模块」章节"
│
├─ 中等变更（新增字段、流程微调等）
│    → AI PM 在飞书文档中起草更新内容，标记为「待确认」
│    → 飞书群发送确认卡片："PR #45 改了登录流程，我建议这样更新 PRD，请确认"
│    → 团队确认后正式写入
│
└─ 重大变更（新功能、架构调整等）
     → AI PM 在飞书群详细说明差异
     → 创建一个「文档更新」讨论串，列出需要更新的所有章节
     → 等待团队讨论后，协助更新

场景 B：每日巡检时触发（定时）
──────────────────────────
每日站会巡检时，额外执行一轮「文档漂移检测」
       ↓
汇总过去 24h 所有合并的 PR
       ↓
逐一对比代码变更与 PRD 内容
       ↓
生成「文档同步报告」（作为每日报告的一部分）：

  📄 文档同步状态
  ├── ✅ 登录模块 — 文档已同步（PR #45 合并后已自动更新）
  ├── ⚠️ 支付模块 — 检测到 3 处差异待确认
  │   ├── 新增了微信支付渠道（代码有，文档无）
  │   ├── 支付超时从 30s 改为 60s（代码已改，文档仍是 30s）
  │   └── 删除了银联支付（代码已删，文档仍有）
  └── ✅ 用户中心 — 无变更

场景 C：手动触发
──────────────────────────
团队成员在飞书群说：
  "@Grove 帮我检查一下文档是不是最新的"
  "@Grove 把最近的代码变更同步到文档里"
       ↓
AI PM 执行全量对比，生成差异报告并逐项更新
```

**文档更新的智能策略：**

AI PM 不是简单地把代码注释搬到文档里，而是以产品经理的视角理解代码变更的产品含义：

| 代码变更 | AI PM 的理解 | 文档更新方式 |
|---------|-------------|-------------|
| 新增 API 字段 `wechat_pay` | 支付模块新增了微信支付渠道 | 在 PRD「支付方式」章节新增微信支付描述 |
| 修改超时常量 `30 → 60` | 支付超时策略调整 | 更新 PRD 中的技术约束说明 |
| 新增组件 `DarkModeToggle` | 产品新增了暗黑模式功能 | 在 PRD 中新增「暗黑模式」功能模块章节 |
| 删除路由 `/api/v1/legacy` | 下线了旧版 API | 在 PRD 中标记对应功能为「已下线」 |
| 重构：提取 `AuthService` 类 | 纯技术重构，产品无变化 | 不更新 PRD（判断为非产品级变更） |

**关键原则：**
- **分级处理** — 小改自动更新，中改草稿确认，大改团队讨论
- **产品视角** — 不照搬代码，而是翻译成产品语言
- **可追溯** — 每次更新都标注来源 PR，方便回查
- **不误改** — 纯技术重构（不影响产品行为的）不会触发文档更新
- **人有最终决定权** — AI PM 建议更新，但团队可以拒绝或修改

**变更感知的技术实现：**

```
代码变更分析 Pipeline：
1. 获取 PR diff（新增/修改/删除的文件和代码行）
2. 过滤掉非产品相关变更（测试文件、CI 配置、代码风格等）
3. 提取产品相关变更（API 路由、UI 组件、配置常量、数据模型等）
4. 用 LLM 将代码变更"翻译"为产品语言描述
5. 与飞书 PRD 文档逐章节语义对比
6. 生成差异列表 + 建议更新内容
7. 根据变更级别选择自动更新或发起确认
```

### 模块 7：团队成员识别与个性化交互（Member Recognition & Personalization）

**核心能力：** AI PM 能分辨团队中的每一个人，记住每个人的角色、负责的工作、沟通偏好，和不同的人聊不一样的内容。

**身份识别机制：**

```
飞书消息进入
       ↓
提取发送者的飞书 Open ID（每条消息自带 sender.id）
       ↓
查询 team.yml，匹配到具体成员

  - 基础信息（角色、技能、权限）

  - 最近的 commit 和 PR 活动
  - 历史交互记录和偏好
       ↓
基于画像生成个性化回复
```

**个性化交互的具体表现：**

**1. 信息密度适配**

```
张三（前端，偏好简洁）问："#23 还差什么？"
→ "差登录表单的错误提示和 loading 状态。API 接口 /api/auth 李四已经 ready。"

李四（后端 Lead，需要全局视图）问同样的问题：

    关联的 #24 注册流程也依赖这个接口，目前还在 todo。这两个 issue 挡住了
    里程碑 3 个后续任务。建议张三本周完成 #23，下周启动 #24。"
```

**2. 主动推送个性化通知**

AI PM 不是等人问才说，而是主动给相关的人推送相关的信息：

| 事件 | 推送给谁 | 推送内容 |
|------|---------|---------|
| 后端 API 接口更新 | 前端（张三、王五） | "李四更新了 `/api/auth` 接口，返回字段新增了 `refresh_token`，可能影响你的登录模块" |
| PR 需要 review | 技能匹配的人 | "@张三 赵六提交了前端部署相关的 PR #46，涉及 Docker 配置，你比较熟悉能帮忙看看吗？" |
| 设计稿更新 | 对应功能的开发者 | "@王五 孙七更新了支付页面设计稿 v3，和你正在做的 #32 相关" |
| 成员连续 2 天无活动 | 该成员（私聊） | "王五，你负责的 #32 支付集成两天没有更新了，有没有遇到什么阻塞？需要协调资源帮你吗？" |
| 里程碑风险 | Lead（李四） | "MVP 里程碑有延期风险：3 个任务可能赶不上。建议方案：A) 加人 B) 缩减范围 C) 延期 3 天" |

**3. 私聊 vs 群聊的行为差异**

| 场景 | AI PM 的行为 |
|------|------------|
| **群聊** @Grove | 回复所有人可见，内容偏概要，不暴露个人工作细节 |
| **私聊** AI-PM | 回复仅对方可见，可以讨论个人任务困难、工作负载、对方案的不同意见等 |
| **群聊提到个人问题** | AI PM 会私聊回复敏感内容（如"你最近工作量是不是太大了"） |

**4. 权限感知**

不同权限级别的成员能让 AI PM 做不同的事：

```yaml
authority_levels:
  owner:          # 项目负责人
    - 所有 member 和 lead 权限
    - 修改 AI PM 配置（巡检时间、通知策略等）
    - 调整里程碑和项目范围
    - 覆盖 AI PM 的任何决策

  lead:           # 技术负责人 / 模块负责人
    - 所有 member 权限
    - 审批任务重新分配

  member:         # 普通成员
    - 查询进度和自己的任务
    - 提出产品想法和需求
    - 接受/拒绝分配给自己的任务
    - 请求任务调整（需 lead 审批）
    - 编辑飞书 PRD 文档
```

```
孙七（member）："@Grove 把 #38 的优先级改成 P0"
AI PM → "@孙七 #38 目前是 P2，调整为 P0 需要 @李四 确认。"

李四（lead）：点击确认卡片 ✅

     → 群通知："#38 优先级已调整为 P0（李四已确认）"
```

```

具体示例（飞书群发起需求）：
```

### 记忆架构

AI PM 的数据分布在三个层面：

**第一层：飞书知识库（产品知识，所有人可见可编辑）**
```
📁 产品空间
├── 📄 产品愿景与定位
├── 📁 PRD 文档/
├── 📄 用户故事总表
├── 📄 里程碑规划
├── 📄 决策记录              ← AI PM 自动维护
└── 📄 变更日志              ← 自动追踪
```

**第二层：GitHub 仓库（代码相关 + 同步副本）**
```
.grove/
├── config.yml                # AI PM 配置（巡检时间、通知偏好等）
├── team.yml                  # 团队成员配置
├── memory/
│   ├── team-profile.md       # 团队成员画像（自动学习）
│   └── daily-snapshots/      # 每日进度快照
│       ├── 2026-03-20.json
│       └── 2026-03-21.json
└── docs-sync/                # 飞书文档的 Markdown 同步副本
    ├── prd-功能A.md
    └── prd-功能B.md
```

**第三层：服务端缓存（运行时数据）**
- 飞书对话上下文（最近 N 条消息，用于理解连续对话）
- GitHub 事件队列（待处理事件）
- LLM 调用缓存（避免重复计算）

**为什么用这种分层架构：**
- 飞书知识库：所有人能直接看到和编辑，非技术成员友好，协作功能最强
- GitHub 仓库：和代码放在一起，PR 对齐检查时无需跨平台读取，有版本历史
- 服务端缓存：运行时数据，不需要持久化，重启后可重建

- [ ] 风险预警逻辑（超期、无活动、PR 积压）→ 飞书 @相关人
- [ ] 飞书群内可回复报告直接触发操作（如 "帮我把 #32 延期"）

**交付物：** 每天早上飞书群自动收到一份可交互的项目状态报告

### Phase 5：PR 需求对齐 + 文档反向同步（第 9-12 周）

**目标：** 代码和文档双向保持一致——PR 检查需求对齐，同时代码变更自动反哺文档

**5a：PR 需求对齐检查（第 9-10 周）**
- [ ] PR Webhook 监听
- [ ] 代码 diff 摘要生成
- [ ] PRD 需求条目匹配算法（从飞书文档读取）
- [ ] 对齐度评估：GitHub PR 评论 + 飞书通知提交者
- [ ] 需求覆盖度追踪
- [ ] 严重偏离时在飞书群 @相关人 + 创建讨论

**5b：代码 → 文档自动反向同步（第 11-12 周）**
- [ ] 代码变更分类引擎（区分产品级变更 vs 纯技术重构）
- [ ] 变更级别判定（小/中/大）→ 自动更新 / 草稿确认 / 团队讨论
- [ ] 飞书 Docs API 写入能力：直接更新 PRD 对应章节
- [ ] 变更标注机制：AI PM 更新的内容带有来源标识（如 "基于 PR #45 自动更新"）
- [ ] 飞书确认卡片：中等变更发送草稿确认，团队一键批准或修改
- [ ] 每日巡检中增加「文档漂移检测」报告板块
- [ ] 变更日志自动写入飞书知识库「变更日志」文档

**交付物：**
- 每个 PR 自动收到产品层面的 review
- PR 合并后，飞书 PRD 文档自动或半自动保持最新
- 每日报告中包含「文档同步状态」板块

### Phase 6：记忆与学习（第 13-14 周）

**目标：** AI PM 越来越"了解"团队

- [ ] 记忆系统实现（`.grove/memory/`）
- [ ] 团队成员画像自动学习（工作习惯、擅长领域）
- [ ] 决策日志自动记录
- [ ] 跨会话上下文保持（飞书对话历史纳入记忆）
- [ ] 进度趋势分析（周报 / 月报，推送飞书 + GitHub）
- [ ] 飞书多维表格进度看板（可选增强）

**交付物：** AI PM 能基于历史数据做更精准的判断和建议，并在飞书中提供可视化进度看板

## 七、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| LLM 幻觉导致错误判断 | 错误的进度评估或任务分配 | 所有操作通过 PR 提交，人工确认后合并 |
| API 成本失控 | 超出预算 | 设置月度 API 调用上限，非关键分析降级到更便宜模型 |
| 团队不信任 AI PM | 形同虚设 | 初期以"建议"为主，逐步建立信任；保持透明 |
| GitHub API 限流 | 功能受阻 | 实现请求缓存和批处理 |
| 需求文档过于复杂 | LLM 无法准确理解全部上下文 | 分块处理，按模块拆分 PRD |
| 文档自动更新出错 | 错误内容写入 PRD | 分级策略：小改自动写入（可回滚），中/大改必须人工确认；飞书文档有修改历史可随时还原 |
| 误判代码变更为产品变更 | 纯重构被当作功能变更更新到文档 | 训练分类模型，维护过滤规则（测试/CI/配置文件排除）；初期偏保守，宁可漏报不误报 |

## 九、冷启动与 Onboarding

AI PM 不只是用于从零开始的新项目，更多时候需要接入一个已经在跑的项目。这个章节定义 AI PM 的"入职流程"。

### 9.1 已有项目接入（AI PM 入职）

```
Step 1：环境准备（10 分钟）
─────────────────────────
- 在 GitHub 仓库安装 AI PM App，授予 Issues / PR / Contents 权限
- 将 AI PM Bot 加入飞书项目群
- 创建 team.yml 配置文件（填写成员信息和飞书 ID 映射）
- 在飞书创建「产品空间」知识库

Step 2：项目扫描（AI PM 自动执行，约 30 分钟）
─────────────────────────
AI PM 会自动执行以下"入职学习"：

  📂 代码仓库扫描
  ├── 读取项目 README、目录结构、package.json / requirements.txt
  ├── 分析最近 100 条 commit 的主题和模式
  ├── 识别主要模块和技术栈
  └── 输出：「项目技术概况」文档

  📋 Issue / PR 历史分析
  ├── 扫描所有 open / recently closed Issues
  ├── 分析标签体系、里程碑结构
  ├── 识别每个成员的历史贡献分布
  └── 输出：「项目现状摘要」+ 初始成员画像

  💬 飞书历史消息扫描（需授权）
  ├── 读取项目群最近 30 天的消息
  ├── 提取讨论过的需求、决策、待办事项
  └── 输出：「需求池初稿」

Step 3：AI PM 生成入职报告（发到飞书群）
─────────────────────────
AI PM 在群里发送一份「入职报告」消息卡片：

  "大家好，我是 Grove，刚完成项目学习，以下是我的理解：

   📊 项目概况：[技术栈、模块结构、核心功能]
   📋 当前状态：[X 个 open Issues，Y 个进行中的 PR]
   👥 团队：[每个人的角色和当前工作]
   📝 我整理的需求池：[从历史讨论中提取]

   请大家看看有没有理解错的地方，直接回复纠正我。
   接下来我会根据这些信息生成初版 PRD 草稿。"

Step 4：PRD 初稿生成 + 团队校准
─────────────────────────
- AI PM 基于扫描结果，在飞书知识库生成 PRD 初稿
- 团队在文档中校准和补充
- 经过 1-2 轮迭代后，PRD 定稿
- AI PM 正式进入日常工作模式
```

**接入完成标志：**
- team.yml 配置完成，所有成员的 GitHub 和飞书 ID 已映射
- 飞书知识库中有至少一份经团队确认的 PRD 文档
- AI PM 能正确识别所有团队成员
- 第一份每日报告成功生成并推送

### 9.2 全新项目冷启动

```
创建 GitHub 仓库
       ↓
安装 AI PM App + 加入飞书群
       ↓
在飞书群和 AI PM 对话：输入产品理念
       ↓
AI PM 引导式提问 → 生成 PRD 初稿
       ↓
团队协作完善 PRD → 定稿
       ↓
AI PM 自动拆解任务 → 创建 Issues → 分配确认
       ↓
进入日常工作模式
```

### 9.3 新成员加入

```
管理员在 team.yml 中添加新成员信息
       ↓
AI PM 检测到配置变更，自动触发 Onboarding
       ↓
飞书私聊新成员：
  "你好 [名字]，我是 Grove，团队的 AI 产品经理。
   我来帮你快速了解项目：

   📋 项目简介：[一段话概述]
   🎯 当前里程碑：[目标 + 截止日期]
   📄 核心文档：[PRD 链接]
   👥 团队成员：[每个人负责什么]

   你的角色是 [role]，目前有 [N] 个待分配的 [role] 相关任务。
   要不要我给你推荐几个适合上手的 Issue？"
       ↓
新成员确认后，AI PM 推荐入手任务
       ↓
后续 AI PM 会重点关注新成员的前两周动态
  - 比普通成员更频繁地检查是否有阻塞
  - 主动提供更多上下文（如关联文档、相关代码位置）
  - 两周后逐渐调整为正常频率
```

## 十、AI PM 人设与沟通风格

### 10.1 基本人设

```yaml
# .grove/config.yml 中配置
persona:
  name: "Grove"                     # 团队内的称呼
  full_name: "Grove PM"             # 正式场合使用
  avatar: "自定义头像"              # 飞书 Bot 头像

  personality:
    base_tone: "专业但不刻板"       # 核心基调
    humor_level: "适度"             # 偶尔轻松但不油腻
    assertiveness: "温和坚定"       # 催进度时不卑不亢
    emoji_usage: "少量"             # 使用但不过度

  principles:
    - "数据驱动，用事实说话"
    - "建议为主，不做强制决策"
    - "保护个人隐私，敏感信息私聊"
    - "承认错误，及时修正"
    - "尊重每个人的专业判断"
```

### 10.2 不同场景的沟通风格

**日常群聊 — 像一个靠谱的同事**

```
❌ 不要这样（太机械）：
"根据数据分析，Issue #32 的进度偏离率为 23.5%，建议采取纠正措施。"

✅ 要这样（像人）：
"#32 支付集成看起来比预期慢了一些，王五你那边是不是卡在什么地方了？需要帮忙的话说一声。"
```

**催进度 — 温和但明确**

```
轻度提醒（第一次）：
"王五，#32 两天没有更新了，进展顺利吗？"

中度提醒（第二次，3天后）：
"王五，#32 已经接近里程碑截止日期了，目前还差 [具体内容]。
 如果有阻塞我可以帮你协调资源，或者我们调整一下排期？"

升级提醒（需要 Lead 介入）：
私聊 Lead："李四，#32 支付集成可能需要关注一下。王五那边
 几天没有动静，里程碑 3 天后截止。你方便了解一下情况吗？"
```

**出错时 — 坦诚认错**

```
"抱歉，我之前把 #38 的优先级判断错了，它应该是 P1 而不是 P2。
 我已经更新了，不好意思给大家带来困扰。"

"刚才自动更新 PRD 的时候，支付模块那段写得不够准确。
 孙七指出了问题，我已经修正了。下次这类变更我会先发确认再写入。"
```

**表达不确定 — 诚实说"不确定"**

```
"这个 PR 改了很多底层逻辑，我不太确定它对用户端的影响范围。
 @李四 你最了解这块，能帮忙看一下吗？"

"从代码来看这个功能可能是用来做 [猜测]，但我不太有把握。
 @王五 你确认一下？我好更新文档。"
```

### 10.3 AI PM 的禁区

AI PM 不应该做以下事情，这些是硬编码的行为约束：

| 禁止行为 | 原因 |
|---------|------|
| 在群里公开批评某个人的代码质量 | 伤害团队关系 |
| 对比两个成员的工作效率 | 制造竞争和对立 |
| 未经确认删除 Issue 或关闭 PR | 可能丢失重要信息 |
| 在群里透露私聊内容 | 侵犯隐私 |
| 给成员"打分"或"排名" | 不是监工，是协作者 |
| 催促非工作时间的进度 | 尊重工作生活边界 |
| 对技术方案做选择 | 产品经理不干预技术决策 |

### 10.4 可定制项

团队可以通过修改 `.grove/config.yml` 来调整 AI PM 的人设：

```yaml
# 沟通语言
language: "zh-CN"                    # 或 "en" / "zh-TW" / 混合

# 催进度的激进程度（1-5）
reminder_intensity: 3                # 1=佛系 3=默认 5=严格

# 每日报告推送时间
daily_report_time: "09:00"           # 飞书群推送时间

# 工作时间范围（非工作时间不催进度、不推送非紧急消息）
work_hours:
  start: "09:00"
  end: "18:00"
  timezone: "Asia/Shanghai"
  workdays: [1, 2, 3, 4, 5]         # 周一到周五

# 是否允许 AI PM 主动发起话题（而非只响应 @）
proactive_messaging: true

# 自动更新文档的激进程度
auto_doc_update_level: "moderate"    # conservative / moderate / aggressive
```
